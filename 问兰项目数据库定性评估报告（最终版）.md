## 问兰项目数据库定性评估报告（最终版）

文档目的

对现有数据库导出 s2b2c_db_20260407192328p0a90.sql 与目标设计文档 README.md 进行最终定性，明确：

1) 现有问题；2) 出错/缺失字段；3) 已造成与将造成的影响；4) 优先整改方向。

------

## 一、最终定性结论（结论先行）

### 结论等级：重大架构不一致（Critical）

现有 SQL 属于“历史 MySQL 业务库结构”，README 属于“目标 Supabase 规范模型 + API 契约”。两者不是局部字段偏差，而是数据模型代际差异，核心体现在：

- 支付链路、退款链路、分佣链路未按 README 的闭环与幂等模型落地；

- 订单状态机未分轨，单字段承载多语义；

- 用户、商家、管理员、代理、佣金规则等主体模型不一致；

- 字段命名、状态枚举、审计追踪、权限边界设计无法直接对齐 API 契约。

定性判断：

当前库可支撑旧业务运行，但不满足 README 目标系统的可靠性与可维护性要求。若继续在现有结构叠加开发，将高概率出现“状态错乱、佣金争议、退款链路不透明、联调反复返工”。

## 四、已造成的影响（含你关心的用户侧）

## 4.1 已可确认影响（高概率正在发生）

1. 支付状态展示不稳：支付结果与订单可见状态可能出现时序不一致。

1. 用户佣金可见性差：手机端难以稳定展示“按层级、按状态、按规则来源”的佣金记录。

1. 退款进度表达不清：申请/审核/执行混在单状态字段，用户感知为“卡住”或“看不懂”。

1. 前后端联调反复改字段：API 契约与库结构错位，导致接口返回字段长期不稳定。

## 4.2 潜在重大影响（若不整改）

1. 财务对账风险：分佣、退款、支付三链路无法形成可审计闭环。

1. 重复计佣/漏计佣纠纷：无标准幂等约束与规则快照。

1. 安全与合规事故风险：导出文件敏感信息暴露。

1. 版本迁移成本失控：继续补丁式开发会放大历史债务。
