# 支付闭环说明

## 1. 当前目标

这份文档不是支付接入教程，而是说明：

**当前项目支付链路已经做到哪里，下一步正式接入时应该遵守什么闭环。**

## 2. 当前 CloudBase 支付入口

当前小程序侧支付入口由以下云函数承接：

- [cloud-mp/cloudfunctions/payment/index.js](C:\Users\21963\WeChatProjects\zz\cloud-mp\cloudfunctions\payment\index.js)

当前 action 包括：

- `prepay`
- `queryStatus`
- `syncWechatPay`
- `notify`

## 2.1 支付环境开关

当前云函数已经支持按环境变量切换支付模式。

建议配置项：

- `PAYMENT_MODE`
- `PAYMENT_PROVIDER`
- `PAYMENT_SIMULATION_AUTO_COMPLETE`
- `PAYMENT_SIMULATION_ALLOW_NOTIFY`
- `PAYMENT_REQUIRE_FORMAL_CONFIG`
- `PAYMENT_WECHAT_APPID`
- `PAYMENT_WECHAT_MCHID`
- `PAYMENT_WECHAT_NOTIFY_URL`
- `PAYMENT_WECHAT_SERIAL_NO`
- `PAYMENT_WECHAT_API_V3_KEY`
- `PAYMENT_WECHAT_PRIVATE_KEY`
- `PAYMENT_WECHAT_PRIVATE_KEY_PATH`
- `PAYMENT_WECHAT_PLATFORM_CERT`
- `PAYMENT_WECHAT_PLATFORM_CERT_PATH`

推荐语义：

- `PAYMENT_MODE=simulation`：迁移期模拟支付
- `PAYMENT_MODE=formal`：正式支付友好模式，未配全会显式拒绝
- `PAYMENT_MODE=disabled`：支付关闭

## 4. 当前闭环语义

### 4.1 `prepay`

职责：

- 校验订单存在且状态可支付
- 生成支付动作所需返回结构
- 迁移期内可在受控条件下将订单推进到待发货

长期要求：

- 正式接入后，`prepay` 只负责生成支付参数
- 不应该在没有回调确认前直接把订单视为最终支付成功
- 正式模式下如果缺少微信支付配置，必须返回明确的 503 错误

### 4.2 `notify`

职责：

- 接收微信支付成功通知
- 通过 `order_no` 找到订单
- 幂等更新订单支付信息和状态

长期要求：

- 回调必须验签
- 回调必须幂等
- 同一订单重复回调不能重复扣库存、重复推进状态
- 没有正式支付配置时，回调必须明确拒绝，而不是静默成功

### 4.3 `queryStatus`

职责：

- 查询当前订单支付结果
- 向前端返回展示层可用的支付状态

### 4.4 `syncWechatPay`

职责：

- 作为支付补偿入口
- 用于前端主动触发“再同步一次支付结果”

长期要求：

- 只能做查询与补偿
- 不应承载非支付类状态篡改
- 不能代替正式支付回调

## 5. 正式支付闭环的推荐形态

### 5.1 交易阶段

1. 创建订单
2. 订单写入 `pending_payment`
3. 调 `prepay`
4. 小程序发起微信支付
5. 微信回调 `notify`
6. 回调成功后推进到 `pending_ship`
7. 前端通过 `queryStatus` / `syncWechatPay` 做展示与补偿

### 5.2 幂等要求

正式接入时必须保证：

- 同一 `order_no` 多次回调，结果一致
- 已支付订单再次回调，不重复更新核心副作用
- 不重复写支付时间
- 不重复写交易号

## 6. 与库存、佣金、售后的关系

支付不是孤立动作，它会影响：

- 订单履约开始
- 发货资格
- 后续退款资格
- 佣金冻结起点或链路前提

因此后面接正式支付时，不能只看支付函数本身，还要同时检查：
