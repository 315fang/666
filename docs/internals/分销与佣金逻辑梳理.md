# 分销与佣金逻辑梳理

## 1. 总体判断

这个项目的分销不是简单的“推广返佣”，而是叠加了：

- 用户上下级链
- 会员 / 团长 / 代理层级
- 直推 / 间推佣金
- 代理发货利润
- 钱包充值与扣减
- 升级缴费
- N 路径货款
- 合伙人退出退款

## 2. 分销关系的基础模型

### 2.1 关系主轴

分销关系核心还是 `User.parent_id` 这条用户自关联链。

同时还有：

- `agent_id`
- `n_leader_id`

这意味着当前系统里至少有三类关系并存：

1. 普通上下级推荐关系
2. 发货归属代理关系
3. N 路径上下级关系

这也是代码复杂的重要来源。

### 2.2 身份层级

从代码看，至少存在这些角色层级：

- 普通用户
- 会员
- 团长
- 代理商
- 合伙人 / 区域代理等扩展层
- N 会员 / N Leader

这些层级既影响：

- 展示价
- 佣金率
- 发货资格
- 钱包使用资格

## 3. 佣金体系

### 3.1 佣金类型

在 `CommissionService` 中，主要佣金类型包括：

- `self`
- `direct`
- `indirect`
- `gap`
- `agent_fulfillment`
- `Stock_Diff`
- 以及扩展的 `n_price_gap`

这意味着佣金系统不只是“推荐返佣”，而是一个综合利润分配系统。

### 3.2 佣金来源

佣金来源大致分为两类：

1. **推荐链佣金**
- 直推
- 间推
- 三级间推

2. **履约与差价佣金**
- 级差利润
- 代理发货利润
- N 路径价差
- 三级代理层间佣金

### 3.3 配置方式

当前支持两套配置口径：

- 商品级佣金配置
- 全局佣金配置

优先级基本是：

1. 商品配置
2. 全局配置
3. 默认值

这也是为什么后台“佣金比例设置”和商品“独立佣金设置”不能混着理解。

## 4. 代理钱包体系

### 4.1 钱包能力

`AgentWalletService` 当前承担：

- 开户
- 充值
- 扣减
- 转账
- 升级缴费入账
- 合伙人退出退款

它本质上已经不是一个简单余额字段，而是一个小型账务系统。

### 4.2 钱包与订单的关系

钱包会参与：

- 订单支付
- 发货扣货款
- 升级缴费
- 充值赠送

所以钱包不只是财务工具，而是交易链路的一部分。

### 4.3 钱包风险

钱包系统里最需要防的是：

- 并发扣款
- 重复回调
- 负余额
- 多事务之间的原子性失效

目前代码已经有事务和加锁意识，但整体上仍是高风险域。

## 5. N 路径逻辑

N 路径是当前项目里最特殊、也最容易把系统复杂度拉爆的一块。

从代码看它至少包括：

- `n_leader_id`
- 小 n / 大 N 的进货价差
- 大 N 给小 n 划拨货款
- N 路径货款申请

本质上，这是一套“特殊代理链 + 专属资金流 + 特殊价差规则”。

如果后续不严格限制边界，这块会持续恶化。

## 6. 升级与退出

### 6.1 升级

升级申请和升级缴费已经纳入正式模型：

- `UpgradeApplication`
- 支付回调后升级费用进入代理钱包

说明升级不是单纯改个角色字段，而是有申请、缴费、审核流程。

### 6.2 退出

`AgentWalletService.processPartnerExit` 已经体现出“合伙人退出退款”的业务规则：

- 退钱包余额
- 退用户余额
- 降级为普通用户
- 写备注与日志

这说明代理体系已经不仅是营销关系，而是带财务责任的身份体系。

## 7. 当前分销域的主要问题

### 7.1 规则太多，集中度还不够

虽然 `CommissionService` 已经集中了一部分规则，但分销逻辑仍然分散在：

- 订单支付
- 发货
- 钱包
- 升级
- 提现
- N 路径

多个 service 和流程里。

### 7.2 身份模型还不够干净

当前角色既影响：

- 价格
- 佣金
- 发货
- 钱包
- 升级资格

这会导致角色变更影响面过大。

### 7.3 财务语义与业务语义混在一起

比如：

- 佣金
- 钱包余额
- 发货利润
- 升级缴费
- 分红池计提

这些都已经是半财务系统语义，但当前仍大量嵌在业务 service 里。

## 8. 我的判断

这个项目的分销域已经不是“商城附加功能”，而是第二主系统。

如果后续要继续做，必须明确边界：

1. 哪些是核心保留能力
2. 哪些是历史扩展能力
3. 哪些必须冻结，不再继续发散

## 9. 建议的保留边界

建议长期保留的最小闭环：

- 直推 / 间推佣金
- 代理发货利润
- 提现
- 钱包充值 / 扣减
- 升级申请

建议后置或严格收缩的部分：

- N 路径复杂规则
- 多层代理层间差价
- 过度复杂的自动分红池策略

## 10. 接手时的建议

1. 分销域所有改动先查 `CommissionService` 和 `AgentWalletService`。
2. 任何新规则都要先判断属于“佣金规则”还是“资金规则”。
3. 不要在 controller 或页面里新增佣金计算。
4. 若要继续做 CloudBase 迁移，分销域必须比商城主链路更晚迁，因为它风险更高。
