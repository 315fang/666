# 字段兼容清理清单

## 1. 目的

这份清单用于约束“哪些旧字段必须被持续清理”，并给出当前优先级。

依据来源：

- [cloud-mp/MYSQL_TO_CLOUDBASE_MAPPING.md](C:\Users\21963\WeChatProjects\zz\cloud-mp\MYSQL_TO_CLOUDBASE_MAPPING.md)
- [cloud-mp/docs/CLOUDBASE_LEGACY_COMPAT_AUDIT.md](C:\Users\21963\WeChatProjects\zz\cloud-mp\docs\CLOUDBASE_LEGACY_COMPAT_AUDIT.md)

## 2. 旧字段到目标字段

| 旧字段 | 目标字段 | 说明 |
| --- | --- | --- |
| `buyer_id` | `openid` | 订单归属统一到 openid |
| `user_id` | `openid` 或后台专用 `admin_id` | 视上下文决定，不再混用 |
| `quantity` | `qty` | 购物车与订单数量统一 |
| `product_skus` | `skus` | 不再保留旧集合名 |
| `nickname` | `nickName` | 展示昵称统一 |
| `avatar_url` | `avatarUrl` | 头像字段统一 |
| `image_url` | `file_id + url` | 长期以 `file_id` 为主 |
| `pending_payment` | 标准订单状态枚举 | 不再扩散旧状态词 |

## 4. 清理优先级

### P0

- `quantity -> qty`
- `image_url -> file_id/url`
- `nickname -> nickName`
- `avatar_url -> avatarUrl`

原因：

- 页面层使用最广
- 直接影响小程序与后台显示
- 清完后兼容层负担会明显下降

### P1

- `user_id`
- `buyer_id`
- `pending_payment`

原因：

- 这类字段更多影响主业务语义和存储层
- 需要在正式数据导入与主链路切换后处理

### P2

- `product_skus`

原因：

- 当前已基本清空，不是最高风险项

## 5. 按区域看清理重点

### 5.1 小程序

优先清：

- `cart`
- `order`
- `distribution`
- `index`

主要问题：

- `quantity`
- `nickname`
- `avatar_url`
- `image_url`

### 5.2 管理后台

优先清：

- `orders`
- `content`
- `home-sections`
- `activities`
- `users`

主要问题：

- `image_url`
- `nickname`
- `avatar_url`
- `quantity`

### 5.3 CloudRun 管理服务

优先清：

- `image_url`
- `pending_ship`
- 个别 `buyer_id` 兼容读取

说明：

- 这一层已经比页面干净很多
- 当前保留兼容更多是迁移期需要

## 6. 清理规则

1. 旧字段只能继续删除，不能新增。
2. 新页面或新接口不得再以旧字段为主字段。
3. 必要兼容必须集中在适配层，不允许散落到页面里。
4. 清理时优先处理展示层，再处理存储层和主链路。

## 7. 推荐执行顺序

1. 小程序 `cart / order / index / distribution`
2. `admin-ui` 的 `orders / content / splash / activities / users`
3. CloudRun 管理服务输出字段继续标准化
4. 正式 CloudBase 导入后，再清 `buyer_id / user_id`

## 8. 完成标准

以下条件满足时，可认为兼容清理基本完成：

1. 页面层不再直接依赖 `quantity`、`nickname`、`avatar_url`、`image_url`
2. 新云函数不再写入旧字段
3. 后台接口不再把旧字段作为主输出
4. 审计脚本结果只剩少量迁移期兜底点
