# 订单交易主链路梳理

## 1. 总体判断

订单链路是这个项目最核心、也最复杂的主链路。

从当前实现看，它已经不是简单的“创建订单 -> 支付 -> 发货”，而是叠加了：

- SKU
- 优惠券
- 积分抵扣
- 拼团
- 砍价
- 限时活动
- 快递 / 自提
- 代理发货 / 平台发货
- 子单拆分
- 钱包支付 / 微信支付 / 零元支付

核心参考代码：

- [backend/services/OrderCreationService.js](C:\Users\21963\WeChatProjects\zz\backend\services\OrderCreationService.js)
- [backend/services/OrderPaymentService.js](C:\Users\21963\WeChatProjects\zz\backend\services\OrderPaymentService.js)

## 2. 下单阶段

### 2.1 输入来源

下单输入支持两种形式：

- `items[]`
- 旧兼容参数：`product_id + sku_id + quantity`

说明当前链路已经兼容“多商品订单”和“旧单商品接口”。

### 2.2 下单前校验

下单前会完成：

- 地址 / 自提点校验
- 商品存在与上下架校验
- SKU 校验
- 库存校验
- 活动互斥校验
- 优惠券适用性校验
- 积分抵扣上限校验

这说明下单逻辑已经非常重，不能再继续堆在 controller 里。

### 2.3 价格确定

价格来源不是一个字段，而是多层计算结果：

1. 用户身份价格
2. SKU 价格
3. 商业策略折扣
4. 拼团 / 砍价 / 限时活动覆盖价
5. 优惠券抵扣
6. 积分抵扣
7. 运费

因此这个项目里“订单金额”一定不能靠页面临时计算，必须以后端为准。

### 2.4 拆单逻辑

`OrderCreationService` 明确支持：

- 代理可发货部分
- 平台可发货部分
- 同一商品拆成父单 + 子单

这意味着订单表虽然结构偏旧，但业务已经进入“履约分流”阶段。

## 3. 支付阶段

### 3.1 支付方式

当前实现支持：

- 微信 JSAPI 支付
- 代理钱包支付
- 零元订单自动支付
- 手动支付补单

代码在 [backend/services/OrderPaymentService.js](C:\Users\21963\WeChatProjects\zz\backend\services\OrderPaymentService.js)。

### 3.2 支付状态推进

核心函数 `_markOrderAsPaid` 会做：

1. 主订单改为已支付
2. 子订单同步改为已支付
3. 自提单生成取货码
4. 用户总消费累计
5. 用户升级逻辑
6. 支付后积分与成长值
7. 分红池计提
8. N 路径差价计提

这说明支付不是单一状态切换，而是整个后续业务副作用的枢纽。

### 3.3 当前风险

支付链路现在最大的问题不是代码缺，而是副作用太多：

- 用户升级
- 钱包变化
- 分红池
- 区域利润
- 购物订单上报
- 佣金后续结算

任何支付状态幂等没处理好，都会引发重复记账风险。

## 4. 履约阶段

### 4.1 履约模式

项目中至少有三种履约状态：

- 平台发货
- 代理发货
- 到店自提

订单里已经存在：

- `fulfillment_type`
- `delivery_type`
- `pickup_station_id`

这说明履约系统已经明显不是单通道。

### 4.2 发货影响

发货不仅是状态流转，还会影响：

- 代理利润
- 中间佣金
- 自提核销信息
- 物流信息

所以“发货”在这个项目里更像结算流程的一部分，不是单纯填写快递单号。

## 5. 售后阶段

### 5.1 退款

退款在当前系统里是独立模型，不是订单状态的附属字段：

- `Refund`
- `Order -> Refund`

这意味着退款链路已经具备独立生命周期。

### 5.2 评价

评价也是独立模型：

- `Review`
- `Order -> Review`
- `Product -> Review`

评价既影响用户可见内容，也影响商品域聚合信息。

## 6. 订单主链路中的核心耦合点

### 6.1 与用户域耦合

- 地址
- 用户等级
- 上下级关系
- 钱包
- 积分

### 6.2 与商品域耦合

- 商品价格
- SKU
- 库存
- 活动商品资格

### 6.3 与分销域耦合

- 直推 / 间推佣金
- 代理发货利润
- N 路径差价

### 6.4 与活动域耦合

- 拼团
- 砍价
- 限时活动
- 优惠券

这就是为什么订单代码很容易变成上帝模块。

## 7. 当前主链路的优点

1. 已经开始 service 化拆分，不是所有逻辑都堆在 controller。
2. 下单与支付主链路已经相对集中，至少能找到入口。
3. 活动单、拆单、钱包支付、自提这些复杂场景已经纳入主链路，不是完全散落。

## 8. 当前主链路的主要问题

1. 金额口径尚未彻底统一到“分”。
2. 订单模型还保留明显的旧单商品结构。
3. 副作用过多，支付幂等与补偿要格外谨慎。
4. 活动规则与主订单模型耦合很深，后续维护成本高。

## 9. 接手时的建议

1. 任何订单改动先看 `OrderCreationService` 和 `OrderPaymentService`。
2. 不要先改页面金额计算，先确认后端口径。
3. 不要把新业务继续塞进订单 controller。
4. 订单状态、履约状态、售后状态必须分开看，不要混成一个维度。
