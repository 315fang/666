# 订单支付售后状态枚举表

## 1. 目的

这个项目当前最容易失真的地方之一，就是状态名。

主要原因：

- 原 MySQL 主系统有一套状态
- CloudBase 迁移线有一套状态
- 前端展示层还有旧别名
- 售后与佣金冻结又依赖订单状态推进

所以这份文档的目标只有一个：

**明确当前各层应该认什么状态。**

## 2. CloudBase 用户侧订单状态

当前 CloudBase 云函数实际存储状态，见：

- [cloud-mp/cloudfunctions/order/index.js](C:\Users\21963\WeChatProjects\zz\cloud-mp\cloudfunctions\order\index.js)
- [cloud-mp/cloudfunctions/payment/index.js](C:\Users\21963\WeChatProjects\zz\cloud-mp\cloudfunctions\payment\index.js)

### 2.1 存储层状态

- `pending_payment`
- `pending_ship`
- `shipped`
- `completed`
- `cancelled`
- `refund_applying`

### 2.2 前端展示兼容状态

当前仍保留以下展示映射：

- `pending_payment` -> `pending`
- `pending_ship` -> `paid`

结论：

- **存储层**继续统一用正式状态
- **展示层**允许兼容 `pending` / `paid`
- 不允许再把展示别名写回数据库

## 4. 推荐长期标准

### 4.1 CloudBase 目标标准

订单正式存储状态建议固定为：

- `pending_payment`
- `pending_ship`
- `shipped`
- `completed`
- `cancelled`
- `refunding`
- `refunded`

说明：

- `pending_payment` 明确优于笼统的 `pending`
- `pending_ship` 明确优于笼统的 `paid`
- `refund_applying` 可作为迁移期兼容状态，长期建议收口到 `refunding`

### 4.2 前端展示状态

前端页面允许继续显示：

- `pending`
- `paid`
- `shipped`
- `completed`
- `cancelled`
- `refunding`
- `refunded`

但这只是 UI 文案层，不是存储层口径。

## 5. 支付状态语义

当前 CloudBase `payment` 云函数语义是：

- `prepay`: 准备支付并进入支付动作
- `queryStatus`: 查询订单支付结果
- `syncWechatPay`: 做支付状态补偿查询
- `notify`: 处理支付回调

当前支付完成后的关键状态推进：

- `pending_payment` -> `pending_ship`

参考：

- [docs/internals/支付闭环说明.md](C:\Users\21963\WeChatProjects\zz\docs\internals\支付闭环说明.md)

补充说明：

- 在 `PAYMENT_MODE=formal` 且配置不完整时，`prepay` 和 `notify` 会显式拒绝，不会伪装成已完成支付。
- 在 `PAYMENT_MODE=simulation` 时，是否自动完成支付由环境变量控制，不能被视为正式支付状态。

## 6. 售后状态

当前退款/售后主状态，见：

- [backend/models/Refund.js](C:\Users\21963\WeChatProjects\zz\backend\models\Refund.js)
- [backend/controllers/refundController.js](C:\Users\21963\WeChatProjects\zz\backend\controllers\refundController.js)

当前核心状态包括：

- `pending`
- `approved`
- `rejected`
- `processing`
- `completed`
- `cancelled`

当前小程序 CloudBase 侧：

- 申请售后时，退款单状态写 `pending`
- 订单状态暂写 `refund_applying`

长期建议：

- 退款单继续保留独立状态流
- 订单仅表达“是否进入退款中/已退款”

## 8. 当前必须遵守的规则

1. 存储层不再新增 `pending`、`paid` 这类模糊状态。
2. 页面展示兼容只能留在展示层。
3. 退款单状态和订单状态必须分开建模。
4. 任何状态新增、重命名、映射调整，都必须同步更新：
   - [docs/internals/订单交易主链路梳理.md](C:\Users\21963\WeChatProjects\zz\docs\internals\订单交易主链路梳理.md)
   - [docs/internals/订单支付售后状态枚举表.md](C:\Users\21963\WeChatProjects\zz\docs\internals\订单支付售后状态枚举表.md)
   - [docs/internals/支付闭环说明.md](C:\Users\21963\WeChatProjects\zz\docs\internals\支付闭环说明.md)
5. 正式支付缺配置时，必须显式失败，不允许回落为假成功。
