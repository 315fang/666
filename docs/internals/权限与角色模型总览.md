# 权限与角色模型总览

## 1. 为什么这份文档重要

这个项目的权限不是简单菜单显隐，而是：

- 后台角色权限
- 接口权限
- 页面路由权限
- 小程序用户身份等级

四套语义混在一起。

如果不先把“角色”和“权限”分开理解，后面任何改动都容易出错。

## 2. 两套角色体系必须分开看

### 2.1 小程序用户角色

这是业务身份，用于决定：

- 看什么价格
- 能拿什么佣金
- 是否可升级
- 是否有代理钱包资格

典型层级包括：

- 普通用户
- 会员
- 团长
- 代理商
- 合伙人 / 区域代理
- N 会员 / N Leader

这套角色主要存在于用户表与业务 service。

### 2.2 后台管理员角色

这是管理身份，用于决定：

- 能否进后台
- 能否看某个模块
- 能否执行管理动作

当前后台角色至少包括：

- `super_admin`
- `admin`
- `operator`
- `finance`
- `customer_service`
- `warehouse`
- `designer`

在 CloudRun 管理服务里，还存在按权限预设映射出来的模块权限。

## 3. 当前权限实现格局

### 3.1 原主系统

权限分散在：

- 后端权限目录 / 中间件
- 管理端路由
- 管理端 store

审计结论已经明确指出：权限名存在漂移，不能视为完全单一真相。

### 3.2 CloudRun 管理服务

在 [backend/cloudrun-admin-service/src/app.js](C:\Users\21963\WeChatProjects\zz\backend\cloudrun-admin-service\src\app.js) 中，已经建立了基于角色预设的权限集合。

当前主要权限模块包括：

- `products`
- `orders`
- `materials`
- `content`
- `settings_manage`
- `order_amount_adjust`
- `order_force_cancel`
- `order_force_complete`

这条线相对更清晰，也更接近长期目标。

## 4. 当前权限问题本质

### 4.1 角色和权限被混用

典型误区是：

- 把 `super_admin` 当权限
- 把 `products` 当角色
- 把用户业务身份和后台管理身份混在一起

这是错误的。

### 4.2 页面权限和接口权限未完全一致

即使某个页面能看见菜单，也不代表接口真正被同名权限保护。

这也是后台长期容易出现“前端能进、接口报错”或“前端看不见、后端却放开”的原因。

### 4.3 CloudBase 迁移期又多了一套权限边界

CloudRun 管理服务已经开始收口权限，但原后台仍有历史逻辑。

因此当前必须接受一个现实：

权限体系还在迁移中，不是已经完全统一。

## 5. 建议的长期权限模型

### 5.1 用户侧

只保留业务身份，不用后台权限语义：

- `role_level`
- `agent_level`
- `purchase_level`

### 5.2 后台侧

明确采用：

- 角色：谁是什么管理员
- 权限：他能做什么

角色只做预设集合，接口永远按权限点判定。

### 5.3 CloudBase 目标

CloudBase 迁移后，建议继续维持：

- `admins`
- `admin_roles`
- `admin_audit_logs`

后台管理员身份不能复用小程序用户身份。

## 6. 当前阶段的正确做法

1. 新后台接口全部走显式权限中间件。
2. 新文档一律区分“业务角色”和“管理角色”。
3. 页面菜单权限和接口权限必须用同一套权限名。
4. 不再新增权限 alias，除非是迁移期兜底。

## 7. 接手时要记住的点

1. 用户等级不是后台管理员角色。
2. 后台角色不是佣金资格。
3. 订单、素材、设置这类高风险动作必须看接口权限，不要只看页面是否显示。
