# 当前业务需求总览

本文档是当前仓库的主业务需求文档。内容依据以下真相源整理：

- 根说明：[`README.md`](../../README.md)
- 收口基线：[`../audit/2026-04-06-repo-audit.md`](../audit/2026-04-06-repo-audit.md)
- 代码入口：`backend/app.js`、`admin-ui/src/router/index.js`、`miniprogram/app.json`

## 1. 当前业务边界

本项目当前真实主系统是一个“微信商城 + 分销代理 + 内容运营 + 活动营销 + 自提履约”的复合系统，主目录只有三条业务主线：

- `backend/`: 用户端 API、管理端 API、定时任务、支付与履约服务
- `admin-ui/`: 运营后台
- `miniprogram/`: 微信原生小程序

同时需要注意两条容易误判的边界：

- `backend` 里仍保留[..](../../../..) `/api/portal/*` 门户接口和门户口令申请链路，但当前仓库里没有可作为主线维护对象的门户前端源码
- `cloud-mp/` 与 `backend/cloudrun-admin-service/` 属于迁移/演进线，不是当前现网主业务的唯一真相源

## 2. 业务定位

这不是普通小商城，当前实现同时覆盖以下目标：

- 面向 C 端用户的小程序商城
- 面向运营、客服、财务、设计的管理后台
- 面向分销和代理的佣金、钱包、发货、提现能力
- 面向 N 路径代理的独立入场、升级、货款协作和价差结算能力
- 面向活动运营的拼团、砍价、抽奖、优惠券、限时活动和内容位运营能力

## 3. 当前真实角色体系

### 3.1 标准路径（C/B）

当前代码中的标准角色是：

- `0`: 普通用户
- `1`: C1 初级代理
- `2`: C2 高级代理
- `3`: B1 推广合伙人
- `4`: B2 运营合伙人
- `5`: B3 区域合伙人

这条路径承接：

- 会员升级
- 分销关系链
- 佣金结算
- 代理发货
- 提现和货款钱包

### 3.2 N 路径

当前代码中的 N 路径不是复用 B2/B3，而是独立角色：

- `6`: 小 n 代理
- `7`: 大 N 独立代理

这条路径承接：

- 邀约入场
- 入场费支付与审核
- 大 N 向小 n 划拨货款
- 小 n 申请货款
- 小 n 升级大 N
- 小 n / 大 N 价差归集与脱离奖励

## 4. 核心业务域

### 4.1 商城交易主链路

小程序当前覆盖的主路径是：

- 首页、分类、搜索、商品详情
- 购物车和直接购买
- 地址管理
- 下单、预支付、支付回调
- 订单列表、订单详情、确认收货
- 评价、退款、物流、自提凭证

当前履约口径不是单一平台发货，而是并存：

- 平台发货
- 代理发货
- 快递配送
- 到店自提

### 4.2 会员、成长与权益

当前用户端和后台都已落地：

- 会员等级与成长值
- 会员中心、成长特权、等级展示
- 签到、积分账户、积分日志、任务
- 收藏、足迹、通知、偏好设置
- 购买价档与成长折扣

### 4.3 分销、代理与资金

当前业务不是“只返佣”，而是包含多条资金链：

- 上下级关系绑定
- 直推、间推、三级佣金
- 商品级佣金与系统级佣金配置
- 佣金流水、结算、提现
- 代理工作台、待发货订单、货款账户、货款充值
- 代理发货利润和发货超时转平台履约

### 4.4 N 路径独立代理

当前已落地的 N 路径能力包括：

- 邀请页和邀约卡
- 加入小 n 的升级申请
- 小 n 升大 N 的两种路径
- 大 N 查看名下小 n
- 大 N 审核小 n 货款申请
- 大 N 主动划拨货款
- 小 n 下单时按自己的拿货价支付，价差归大 N

### 4.5 营销活动与内容运营

当前仓库已实现的活动和内容域包括：

- 拼团
- 砍价
- 抽奖
- 优惠券
- 限时活动
- Banner、图文内容、素材库
- 商品推荐榜、首页内容位、页面编排、开屏动画

活动并不是独立结算体系，仍然会落回商品、订单、库存、支付和售后主链路。

### 4.6 后台运营与治理

管理后台当前真实职责包括：

- 商品、分类、订单、退款、物流、自提门店
- 用户、经销商、分支代理、N 路径代理
- 会员与成长、代理体系、运营参数
- 内容、素材、推荐位、首页内容位、群发消息
- 佣金、提现、日志、管理员与权限

## 5. 关键业务流程

### 5.1 普通用户流程

1. 进入小程序首页或分类页
2. 浏览商品、活动和内容位
3. 加入购物车或直接下单
4. 选择配送方式、自提点、优惠券和积分
5. 完成支付
6. 进入发货、物流、自提、确认收货、评价或退款流程

### 5.2 标准分销/代理流程

1. 用户通过邀请或升级进入标准角色路径
2. 下级成交后，上级按当前佣金规则获得收益
3. 满足条件的代理可承接发货并获得履约利润
4. 收益进入佣金或货款账户，后续用于提现、发货或继续经营

### 5.3 N 路径流程

1. 大 N 通过邀约卡邀请用户加入
2. 用户提交 `n_join` 申请并完成支付/审核
3. 成为小 n 后，由大 N 提供货款额度
4. 小 n 用货款下单提货，价差自动归集到大 N
5. 小 n 满足条件后可通过付费或团队路径升级为大 N

### 5.4 后台运营流程

1. 运营配置商品、价格档、活动和内容位
2. 客服/仓储/财务在后台处理订单、退款、发货、提现和权限
3. 代理体系与会员成长策略通过后台配置页调整
4. 业务口径变化必须同步文档、后台配置和服务层实现

## 6. 当前不应误判的事项

- 当前现网主业务仍以 `backend + admin-ui + miniprogram` 为准
- 门户后端接口仍存在，但不应把不存在的 `agent-portal/` 当成当前仓库主前端
- CloudBase 相关目录是迁移线，不应覆盖当前主业务事实
- 历史阶段设计稿、旧计划、旧手册不再作为当前需求依据

## 7. 当前最重要的约束

1. 这是复合业务系统，任何订单、佣金、钱包、活动改动都可能跨域影响
2. 需求变更必须先明确是标准路径、N 路径还是内容/活动域，不可混写
3. 当前阶段优先收口真实业务口径，而不是继续叠加未经梳理的新玩法
