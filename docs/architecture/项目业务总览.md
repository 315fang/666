# 项目业务总览

## 1. 项目是什么

这是一个微信电商分销系统，不是单纯商城。它同时包含：

- C 端小程序商城
- 管理后台
- 后端 API 与定时任务
- 分销与代理体系
- 活动营销体系
- 内容与素材体系

## 2. 主要业务域

### 2.1 用户与身份

核心对象：

- 用户
- 地址
- 收藏
- 通知
- 会员等级
- 上下级关系
- 代理钱包

业务目标：

- 识别用户身份与层级
- 管理收货地址和个人资料
- 建立分销关系链
- 承接积分、佣金、提现、升级等后续能力

### 2.2 商品与内容

核心对象：

- 商品
- 分类
- SKU
- Banner
- 内容页
- 页面装修板块
- 素材库

业务目标：

- 提供商品销售与类目组织
- 给小程序首页、活动页、分类页提供可配置内容位
- 让后台运营可上传素材、配置 Banner 和图文内容

### 2.3 交易主链路

核心对象：

- 购物车
- 订单
- 支付
- 发货
- 收货
- 退款
- 评价

业务目标：

- 完成从下单到售后的闭环
- 支持快递与到店自提
- 支持拆单、子单、代理发货、平台发货等履约模式

### 2.4 分销与代理

核心对象：

- 直推 / 间推佣金
- 代理商发货利润
- 提现
- 代理钱包
- N 路径货款

业务目标：

- 让用户关系链具备收益分配能力
- 支持代理发货和平台发货并存
- 支持升级申请、货款充值、钱包扣减

### 2.5 营销活动

核心对象：

- 拼团
- 砍价
- 抽奖
- 优惠券
- 限时活动

业务目标：

- 提供增长和促活能力
- 与交易主链路结合，形成特定的价格、库存、资格规则

### 2.6 运营与系统治理

核心对象：

- 管理员
- 权限
- 系统配置
- 群发
- 告警
- 操作日志
- 统计

业务目标：

- 让运营、客服、财务、设计等后台角色可以工作
- 对交易与内容系统进行可追踪的管理

## 3. 用户视角下的业务流程

### 3.1 普通商城路径

1. 用户进入小程序
2. 浏览首页、分类、搜索、商品详情
3. 加入购物车或直接下单
4. 选择地址 / 配送方式 / 自提点
5. 使用优惠券或积分
6. 支付订单
7. 等待发货、收货、评价

### 3.2 分销路径

1. 用户通过邀请关系成为下级
2. 根据身份等级获取不同商品价或佣金资格
3. 上级在下级成交时获得直推 / 间推收益
4. 代理商可能承担发货职责并获取发货利润
5. 收益进入佣金或钱包体系，后续可提现或继续使用

### 3.3 活动路径

1. 用户参与拼团 / 砍价 / 抽奖 / 限时活动
2. 活动对商品价格、库存、资格产生附加限制
3. 最终仍回落到订单、支付、履约、退款主链路

## 4. 当前项目的真实复杂点

这个项目难，不是因为页面多，而是因为几条线叠加：

- 商城交易线
- 分销代理线
- 钱包资金线
- 营销活动线
- 内容运营线
- CloudBase 迁移线

这意味着后端和数据库天然复杂，前端页面也容易承担过多展示兼容逻辑。

## 5. 当前最关键的工程事实

### 5.1 主业务已经做出来

从模型和服务数量看，项目不是空壳。尤其订单、支付、佣金、钱包、素材、营销这些核心对象都存在。

### 5.2 业务逻辑比工程治理成熟

目前的主要问题不是业务不存在，而是：

- 真相源不单一
- 字段命名不统一
- CloudBase 迁移与原系统并存
- 页面层兼容逻辑太多

### 5.3 当前最需要的是收口，不是继续堆功能

如果继续新增功能，而不先统一：

- 数据模型
- 权限模型
- 支付口径
- 订单状态口径
- 素材字段口径

项目维护成本会继续快速上升。

## 6. 接手项目时应该先记住的三件事

1. 这是“电商 + 分销 + 钱包 + 活动 + 内容”的复合系统，不是普通小商城。
2. 真正的风险在“跨域耦合”，尤其是订单、佣金、钱包、活动之间的连锁影响。
3. 当前最重要的工作不是重写，而是恢复单一真相源并收口兼容层。
