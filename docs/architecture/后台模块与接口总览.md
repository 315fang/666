# 后台模块与接口总览

## 1. 后台当前不是单块系统

当前后台实际上由两部分组成：

- [admin-ui](C:\Users\21963\WeChatProjects\zz\admin-ui)：Vue 管理前端
- [backend](C:\Users\21963\WeChatProjects\zz\backend) 或 [backend/cloudrun-admin-service](C:\Users\21963\WeChatProjects\zz\backend\cloudrun-admin-service)：后端接口层

也就是说，后台逻辑当前也有“原主系统”和“CloudBase 迁移新底座”两条线。

## 2. 管理前端模块

从 [admin-ui/src](C:\Users\21963\WeChatProjects\zz\admin-ui\src) 看，后台前端主要包含：

- 商品管理
- 订单管理
- 内容管理
- 活动管理
- 用户管理
- 分销 / 佣金 / 提现
- 系统设置
- 物流 / 自提 / 服务站
- 素材与装修

这说明后台不是单纯电商后台，而是“商城 + 分销 + 内容 + 运营后台”。

## 4. 当前后台模块边界

### 4.1 适合继续往 CloudRun 收口的模块

- 商品
- 分类
- 素材
- Banner / 内容
- 订单
- 设置

这些模块边界相对清晰，而且已经有新底座。

### 4.2 暂不适合优先迁移的模块

- 复杂分销看板
- 深层财务报表
- 营销活动高级后台
- 历史 MySQL 特化工具

原因是这些模块规则重、依赖旧数据深、迁移风险高。

## 5. 当前后台接口最需要注意的事

### 5.1 后台不应直写数据库

长期目标必须是：

- 前台页面只调 API
- 所有写操作都经过服务层

### 5.2 后台接口不应复用小程序用户侧接口

用户侧云函数和后台管理接口是两套边界，不应该混为一谈。

### 5.3 后台接口输出必须逐步标准化

尤其需要逐步统一：

- `id`
- `file_id`
- `nickName`
- `avatarUrl`
- `qty`

而不是继续把旧字段当主字段输出。

## 6. 当前阶段推荐的后台阅读顺序

1. 先看 [admin-ui](C:\Users\21963\WeChatProjects\zz\admin-ui) 里的页面模块划分
2. 再看 [backend/cloudrun-admin-service/src/app.js](C:\Users\21963\WeChatProjects\zz\backend\cloudrun-admin-service\src\app.js) 的新底座能力
3. 最后再看原 [backend/routes/admin](C:\Users\21963\WeChatProjects\zz\backend\routes\admin) 的历史主系统接口

## 7. 长期推荐形态

长期最优解应是：

- `admin-ui` 保留现有前端技术栈
- CloudRun 提供后台专用 API
- 小程序用户侧继续走云函数
- 后台与小程序共用标准数据模型，但不共用身份体系
