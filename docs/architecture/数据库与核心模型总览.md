## 3. 关键关联关系

从 [backend/models/index.js](C:\Users\21963\WeChatProjects\zz\backend\models\index.js) 看，最重要的关联有：

### 3.1 用户与订单

- `User -> Order` 通过 `buyer_id`
- `User -> Order(distributedOrders)` 通过 `distributor_id`
- `User -> User` 自关联承载 `parent_id`

这说明当前分销关系核心仍然绑在用户上下级链上。

### 3.2 商品与订单

- `Product -> Order`
- `Product -> SKU`
- `Order -> SKU`
- `Cart -> SKU`

这说明当前系统虽然支持 SKU，但很多逻辑仍保留“订单直接挂商品和单一 SKU”的旧结构。

### 3.3 订单与佣金

- `Order -> CommissionLog`
- `CommissionSettlement -> CommissionLog`

说明佣金不是简单字段，而是独立流水体系。

### 3.4 订单与退款 / 评价

- `Order -> Refund`
- `Order -> Review`

说明售后和评价都已经独立建模。

### 3.5 素材与内容

- `Material -> Product`
- `MaterialGroup -> Material`
- `Banner -> Product`
- `ContentBoard -> ContentBoardProduct -> Product`

说明内容与商品之间已经建立了可运营化关系，但图片长期字段仍未完全统一。

## 4. 当前数据模型的主要问题

### 4.1 字段语言分裂

当前同时存在两套字段语言：

- MySQL 旧字段：`buyer_id`、`user_id`、`quantity`、`nickname`、`avatar_url`
- CloudBase 目标字段：`openid`、`qty`、`nickName`、`avatarUrl`、`file_id`

这是当前兼容代码大量存在的直接原因。

### 4.2 订单模型偏旧

当前 MySQL 主模型仍偏单商品订单视角，虽然已支持拆单和子单，但长期仍应朝：

- `items[]`
- 订单快照
- 金额分摊
- 售后明细

这样的结构演进。

### 4.3 图片字段未统一

当前图片仍散落在：

- `images`
- `detail_images`
- `image_url`
- `url`

长期应该统一成：

- `file_id`
- 必要时派生 `temp_url`

### 4.4 金额单位未完全统一

当前系统仍大量以“元”为主运算，CloudBase 目标文档已经明确长期应收口到“分”。

这是一项必须做的基础治理，否则支付、优惠券、积分、佣金都容易积累精度问题。

## 5. CloudBase 目标模型

迁移目标集合主要是：

- `users`
- `products`
- `skus`
- `categories`
- `cart_items`
- `orders`
- `refunds`
- `reviews`
- `commissions`
- `withdrawals`
- `configs`
- `banners`
- `materials`
- `material_groups`
- `admins`
- `admin_roles`

目标原则是：

1. MySQL 是迁移输入，不是长期运行标准。
2. CloudBase 新代码只认新字段。
3. 兼容逻辑只允许出现在迁移脚本和临时适配层。

## 6. 接手时应该如何理解数据库

建议按三层理解：

### 第一层：生产真相

以 [backend/models/index.js](C:\Users\21963\WeChatProjects\zz\backend\models\index.js) 和对应 Sequelize 模型为准。

### 第二层：迁移映射

以 [cloud-mp/MYSQL_TO_CLOUDBASE_MAPPING.md](C:\Users\21963\WeChatProjects\zz\cloud-mp\MYSQL_TO_CLOUDBASE_MAPPING.md) 为准。

### 第三层：兼容代码

任何同时读写旧字段和新字段的地方，都应视为“迁移期代码”，不是最终形态。
