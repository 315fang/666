## 1. 文档目标

本文件用于统一以下两份文档的业务口径，并作为后续数据库字段设计、表结构调整、后端接口实现与状态字段拆分的依据：

1. `docs/architecture/项目业务总览.md`
2. 项目级需求文档主对比对象：`/Users/ury/Downloads/project/问兰微信小程序/prod/需求文档(总).md`

本文件重点解决以下问题：

- 统一项目业务边界、角色体系、订单链路、分佣链路、升级链路的术语和数据表达
- 梳理两份文档中对数据库字段设计有影响的一致点、冲突点和缺失点
- 明确哪些对象应直接落表，哪些应作为配置化规则或待确认扩展域处理
- 形成可执行的命名规范、状态字段拆分规范和冗余兼容策略

## 2. 统一后的业务范围

项目统一后的真实业务边界定义如下：

- 本项目不是普通单体商城，而是“微信商城成交 + 分销代理成长 + 内容活动运营 + 自提/门店履约 + 后台规则管理”的复合业务系统
- 当前现网主系统以 `backend + admin-ui + miniprogram` 为准
- 当前数据库设计必须优先承接以下核心链路：
  - 用户注册/登录与身份升级
  - 商品展示、SKU、库存、价格与分佣配置
  - 下单、支付、支付回调、履约、收货、退款
  - 标准路径分佣、代理发货利润、N 路径价差与提现
  - 自提门店/服务站点承接的线下履约
  - 后台审核、配置、日志与规则管理
- 需求文档中出现的“资源模式、评级奖、3800/9万等合作档位、区域经营奖励”属于强业务影响域，但当前文档口径与现网实现未完全对齐，不能直接硬编码进核心用户表枚举，需通过独立申请/规则/配置表承接

### 2.1 术语映射表

| 需求文档术语 | 项目业务总览术语 | 统一术语 | 数据库表达建议 |
|---|---|---|---|
| 普通消费者 | 普通用户 | 普通用户 | `users.role_level = 0` |
| 会员 | C1 初级代理 | 标准路径一级身份 | `users.role_level = 1` |
| 团长 / 更高等级如 C2 | C2 高级代理 | 标准路径二级身份 | `users.role_level = 2` |
| 基础参与者 / B1 | B1 推广合伙人 | 标准路径三级身份 | `users.role_level = 3` |
| 运营合伙人 | B2 运营合伙人 | 标准路径四级身份 | `users.role_level = 4` |
| 区域合伙人 / 区域代理 | B3 区域合伙人 | 标准路径五级身份 | `users.role_level = 5`，区域归属另表/另字段承接 |
| 小 n / 大 N | 小 n / 大 N | N 路径身份 | `users.role_level = 6/7`，并用 `n_leader_user_id` 表达关系 |
| 实体门店 / 自提点 / 线下服务点 | 自提门店 | 线下站点/门店实体 | 不建议直接并入 `users.role_level`，优先独立为 `service_stations` / `dealers` |
| 团队模式 / 资源模式 | 未单独命名 | 业务模式 | 不建议直接作为用户主角色枚举，可作为申请/规则/活动标签 |
| 评级奖 / 推荐奖 | 未出现在项目业务总览 | 资格类奖励 | 不应混入订单退款单，需独立规则与结算承载 |

### 2.2 统一后的核心链路

1. 用户以 `users` 为唯一主体进入系统
2. 用户在标准路径或 N 路径下形成身份和上下级关系
3. 商品、SKU、库存、价格、佣金配置驱动下单与支付
4. 支付成功后进入履约、自提/快递、收货与退款链路
5. 订单相关收益进入分佣、发货利润或价差结算链路
6. 身份升级、资格生效、犹豫期/冻结期与后台审核形成第二条资格链路

## 3. 两份文档冲突清单

### 3.1 已一致的内容

- 两份文档都明确项目不是普通商城，而是包含分销/成长/运营的复合业务系统
- 两份文档都承认线上成交、自提/线下履约、后台规则管理是核心组成
- 两份文档都将订单、支付、退款、收益/奖励视为核心链路
- 两份文档都默认存在身份升级、团队关系或代理关系
- 两份文档都要求后台能管理商品、订单、用户和奖励规则

### 3.2 冲突清单

| 冲突编号 | 主题 | 项目业务总览口径 | 需求文档口径 | 冲突说明 | 建议采用口径 | 对数据库设计的影响 |
|---|---|---|---|---|---|---|
| C01 | 项目主边界 | 现网主系统是微信商城、分销代理、内容活动、自提履约、N 路径 | 项目是线上卖货、线下门店、自提点、区域经营、资源招商、评级奖综合平台 | 需求文档覆盖更广，包含较多战略扩展域；项目业务总览强调当前现网主线 | 数据库主干先以现网已承接链路为核心，线下/资源/区域作为可扩展域独立建模 | 不能把资源招商、区域经营直接塞进用户主表或订单主表；应抽离申请表、规则表、归属表 |
| C02 | 身份体系定义 | 明确标准路径 `0~5` 与 N 路径 `6~7` | 以普通消费者、B1、运营合伙人、门店、自提点、区域代理、资源合作方描述 | 需求文档角色名称与现网枚举不完全对应，且门店/自提点是否为用户角色不清 | 用户主体仍统一为 `users`，标准路径与 N 路径以角色等级表达；门店/自提点不默认并入主角色 | 角色枚举与门店/站点实体必须拆开；避免把线下站点直接编码成 `role_level` |
| C03 | N 路径是否为主业务 | 明确存在小 n / 大 N 独立路径 | 完全未提 N 路径 | 一份文档有、另一份缺失，但 N 路径已影响当前现网订单与升级链路 | 保留 N 路径为当前必须承接域 | `users` 需保留 N 路径关系字段；升级申请需支持 `path_type` |
| C04 | 资源模式 / 评级奖 | 未作为当前主实现写入 | 明确强调资源模式、评级奖、20% 推荐奖 | 需求文档把资源型招商作为重要业务，但项目业务总览未纳入现网主口径 | 作为扩展资格奖励域保留，不纳入当前核心角色枚举 | 需要预留资格奖励规则表/资格申请表/延迟结算字段；不能直接写死到 `commission_logs` 主规则 |
| C05 | 线下门店 / 自提点定位 | 仅明确“自提门店”与订单配送方式 | 强调实体门店、自提点、区域服务点是重要业务实体 | 一份文档偏现网履约节点，一份文档偏线下经营实体 | 统一为“线下站点/门店实体”，先支持自提履约，经营支持另行扩展 | 建议保留 `service_stations` 与成员表；是否需要独立商家账户待确认 |
| C06 | 区域代理与区域奖励 | 仅出现“区域合伙人”角色名，未定义区域奖励机制 | 明确要求区域奖励、区间递增、区域经营和区域归属 | 需求文档存在强规则，但项目业务总览未给出现网实现细则 | 区域奖励不进入用户主表枚举，按独立规则与归属域处理 | 需要 `region_rule_sets`/归属字段或区域表；当前不足以直接硬编码到 `users` |
| C07 | 分佣口径 | 当前业务包含直推、间推、三级佣金、商品级佣金、N 路径价差、代理利润 | 平台所有奖励统一采用百分比，不采用固定金额作为制度基础 | 与当前已存在的固定金额类奖励、商品级固定金额口径冲突 | 分佣规则引擎必须同时支持比例和固定金额两种模式 | `commission_rules` 需包含 `calculation_mode`、`rate`、`fixed_amount`，不可只保留比例字段 |
| C08 | 高档位金额体系 | 当前角色等级以 C/B/N 路径为主，未把 3800/9万写入主身份口径 | 明确提出 3800、3万、9万、198000 等合作档位 | 需求文档强调金额档位，现网总览强调角色等级 | 金额档位不直接写死为 `users.role_level`，应进等级配置或申请记录 | 需增加等级配置表或在升级申请中保存 `tier_code`、`application_amount` |
| C09 | 退款域边界 | 订单售后退款为现网主链路 | 除订单退款外，还提出高档位合作的犹豫期退款和手续费 | 订单退款与资格退款不是同一业务对象 | 将订单退款与资格/合作退款拆为两类数据域 | `refunds` 只承接订单售后；资格退款建议挂到升级/合作申请域或新增资格退款表 |
| C10 | 订单状态表达 | 文档只描述多阶段链路，未给单一状态方案 | 需求文档描述订单、退款、收益可视化，但未明确字段拆分 | 两份文档都不足以支撑“单状态字段” | 为承接支付、履约、自提、退款、分佣，订单状态必须拆分 | `order_status`、`payment_status`、`fulfillment_status`、`refund_status`、`commission_status` 不可混用 |
| C11 | 商家/门店是否独立后台 | 当前只明确平台后台、代理/N 路径相关后台能力 | 需求文档描述门店/自提点/区域代理管理，但未明确独立商家后台 | 是否需要 `merchant_accounts` 不明确 | 当前不把门店/站点直接建成商家后台模型，待确认后再加 | `merchant_accounts`、`merchant_members` 标记待确认；当前更接近 `dealers` / `service_stations` / `station_staff` |
| C12 | 用户关系字段 | 现网口径强调上下级关系、N 路径关系 | 需求文档强调团队归属、区域归属、门店归属，但未给字段表达 | 需求文档关系多但表达抽象 | 统一采用数值型主键关系字段，不用邀请码/编号做主关系 | 新设计中关系字段统一为 `*_user_id`、`*_station_id`、`*_dealer_id` |

### 3.3 一方有、另一方缺失但会影响数据库设计的内容

- 仅项目业务总览明确了 N 路径，这要求数据库不能只设计标准分销链
- 仅需求文档明确了“资格类奖励、犹豫期退款、区域奖励、自提服务费”，这意味着数据库不能只设计订单佣金
- 两份主对比文档都没有把“支付单是否独立、支付回调是否独立、状态字段是否拆分”写清，这部分必须在数据库设计层补齐
- 两份主对比文档都没有正式定义“商家账户/商家成员”模型，不能直接按多商户平台建核心关系

### 3.4 对当前数据库设计的动作判定

| 编号 | 设计问题 | 来源判断 | 建议动作类型 | 说明 |

## 4. 统一后的核心业务对象

### 4.1 用户 `users`

- 业务职责：
  - 作为系统唯一自然人主体，承接登录、身份、上下级关系、余额、邀请码、成长值、参与分销资格
- 与其他对象关系：
  - 与订单是买家关系
  - 与分佣日志是受益人关系
  - 与升级申请、提现申请、退款申请、N 路径申请有关
  - 与门店/站点、经销商、区域归属是附属关系，不应被线下实体替代
- 对数据库表设计的要求：
  - `users.id` 为唯一主键
  - 标准路径上下级用 `parent_user_id`
  - N 路径上下级用 `n_leader_user_id`
  - 邀请码、会员编号只能做展示或绑定辅助，不能作为主关系外键
  - 用户角色只表达“人”的业务身份，不直接承载门店/自提点实体属性

### 4.2 用户等级/代理等级 `agent_levels` 或 `role_levels`

- 业务职责：
  - 承接标准路径、N 路径和未来资格档位的等级定义、门槛、权益、购买价档和是否可参与分佣
- 与其他对象关系：
  - `users.role_level` 或 `users.level_code` 指向等级定义
  - 升级申请、分佣规则、价格策略依赖等级配置
- 对数据库表设计的要求：
  - 不建议把所有等级门槛写死在代码常量
  - 至少要支持：标准路径等级、N 路径等级、是否参与分佣、默认价格档、是否允许发货/提现/代理功能
  - 如保留数值型 `role_level`，仍建议增加等级配置表作为解释层

### 4.3 商品 `products`

- 业务职责：
  - 承接商品销售、主推标记、活动归属、价格档位、分佣配置和履约方式
- 与其他对象关系：
  - 与 SKU、库存、商品图片、订单、活动、内容位有关
- 对数据库表设计的要求：
  - 商品主表至少包含：名称、类目、主图、详情图、零售价、会员价/团长价/代理价、总库存、销售状态、是否支持自提、是否主推/活动商品
  - 商品是否绑定商家/门店：当前两份文档都未明确，不能强制增加 `merchant_id`
  - 商品级佣金与全局佣金并存时，商品表需保留“是否启用商品级规则”和对应的比例/固定金额字段，或迁移至规则表

### 4.4 商品图片 / SKU / 库存 `product_images`、`product_skus`、库存流水表

- 业务职责：
  - 分离商品主资料、规格、图集与库存变动
- 与其他对象关系：
  - SKU 与订单项、购物车、库存扣减、活动库存强关联
- 对数据库表设计的要求：
  - `product_images` 负责多图和详情图扩展，不建议把所有图片都继续挤在商品主表文本字段里
  - `product_skus` 负责规格、规格价、规格库存、SKU 图片
  - 需要库存流水/预占结构时，优先使用独立表，不把所有库存事件塞进 `products.stock`

### 4.5 订单 `orders`

- 业务职责：
  - 承接下单主体、价格快照、配送方式、履约归属、优惠券积分抵扣和订单生命周期
- 与其他对象关系：
  - 关联用户、商品/SKU、支付记录、退款单、分佣记录、自提门店
- 对数据库表设计的要求：
  - 至少记录：买家、商品/规格快照、数量、下单时价格快照、配送方式、收货/自提信息、参与分佣的上游角色信息
  - 不允许继续用一个字段同时表示“买家归属代理”和“实际履约代理”
  - 为兼容自提/快递/代理发货，需要将订单状态、支付状态、履约状态拆开

### 4.6 支付记录 `payment_records`

- 业务职责：
  - 承接订单支付、升级申请支付、钱包充值等支付业务的统一流水
- 与其他对象关系：
  - 可关联订单、升级申请、钱包充值单等业务对象
  - 与支付回调是一对多
- 对数据库表设计的要求：
  - 建议独立于订单主表
  - 至少支持：`biz_type`、`biz_id`、`channel`、`out_trade_no`、`provider_trade_no`、`payment_amount`、`payment_status`
  - 同一支付平台订单号必须唯一
  - 允许一套支付表同时承接订单支付与升级支付，避免为每条业务链重复造轮子

### 4.7 支付回调 `payment_callbacks`

- 业务职责：
  - 记录微信等支付网关回调原文、验签结果、处理结果和幂等状态
- 与其他对象关系：
  - 关联 `payment_records`
- 对数据库表设计的要求：
  - 建议独立保存原始回调
  - 必须支持幂等处理标记，避免重复改单、重复分佣、重复写账

### 4.8 退款单 `refunds`

- 业务职责：
  - 承接订单售后退款、退货退款、换货等售后申请
- 与其他对象关系：
  - 关联订单、申请用户、审核管理员
- 对数据库表设计的要求：
  - 订单售后退款必须独立成表，不能只在订单表上放一个退款状态
  - 至少记录：退款原因、说明、图片凭证、退款金额、退款数量、退款状态、审核人、驳回原因、处理时间
  - 资格类退款不建议混入本表

### 4.9 分佣记录 `commission_logs` / `commission_records`

- 业务职责：
  - 承接订单分佣、代理发货利润、N 路径价差、资格奖励等收益明细
- 与其他对象关系：
  - 关联订单、受益用户、规则集或规则快照
- 对数据库表设计的要求：
  - 分佣必须独立成表
  - 分佣状态不能与订单状态混用
  - 需要支持多种收益类型：直推、间推、三级、代理利润、N 路径价差、资格类奖励等

### 4.10 分佣规则/规则集 `commission_rule_sets`、`commission_rules`

- 业务职责：
  - 承接按等级、按商品、按路径、按区域、按门店、自提服务费等规则配置
- 与其他对象关系：
  - 与用户等级、商品、区域、门店、分佣记录相关
- 对数据库表设计的要求：
  - 规则集负责版本、场景、启停、生效时间
  - 规则明细负责收益对象、层级、计算方式、条件和优先级
  - 必须支持比例与固定金额双模式，否则无法兼容现网与需求口径差异

### 4.11 自提门店 / 服务站点 `service_stations`

- 业务职责：
  - 承接线下自提、门店展示、站点信息与可能的站点收益
- 与其他对象关系：
  - 与订单自提、门店成员、站点认领、站点补贴相关
- 对数据库表设计的要求：
  - 站点是独立实体，不建议直接用用户角色代替
  - 至少包含站点名称、地址、联系人、营业信息、状态、是否支持自提

### 4.12 门店成员 `station_staff`

- 业务职责：
  - 承接门店/站点下的成员、店长、核销权限
- 与其他对象关系：
  - 关联站点与用户
- 对数据库表设计的要求：
  - 采用 `station_id + user_id` 唯一约束
  - 不建议把门店成员直接混入管理员表

### 4.13 经销商 / 线下合作主体 `dealers`

- 业务职责：
  - 承接经销商、公司主体、结算方式和合作审核
- 与其他对象关系：
  - 关联用户、可能的线下合作主体
- 对数据库表设计的要求：
  - 当前更接近“线下合作主体”而不是“平台多商户”
  - 与 `merchant_accounts` 不应混用命名

### 4.14 升级申请 / 资格申请 `upgrade_applications`

- 业务职责：
  - 承接标准路径升级、N 路径加入/升级以及未来合作档位申请
- 与其他对象关系：
  - 关联用户、目标等级、支付记录、审核管理员、邀请人/上级
- 对数据库表设计的要求：
  - 升级申请必须独立成表
  - 至少支持：申请路径、当前等级、目标等级、申请金额、支付状态、审核状态、邀请关系、是否团队升级
  - 若需求文档中的犹豫期资格退款正式落地，应在本域增加资格生效、犹豫期起止、退款窗口或独立退款表

### 4.15 商家账户 `merchant_accounts`（待确认）

- 业务职责：
  - 仅在项目明确采用“多商户/商家独立后台”模式时成立
- 与其他对象关系：
  - 若落地，应与商品、订单、商家成员强关联
- 对数据库表设计的要求：
  - 当前两份主对比文档都未明确平台采用多商户模式
  - 现阶段不建议把 `dealers`、`service_stations` 直接命名为 `merchant_accounts`

### 4.16 商家成员 `merchant_members`（待确认）

- 业务职责：
  - 仅在 `merchant_accounts` 成立时用于承接商家后台成员
- 与其他对象关系：
  - 关联商家账户与用户
- 对数据库表设计的要求：
  - 当前不作为必落表
  - 若后续需要商家后台，再与平台管理员、门店成员做边界拆分

## 5. 统一后的关键业务规则

### 5.1 身份与上下级关系规则

- 用户是唯一自然人主体，所有分销、代理、门店成员、N 路径关系都从用户延伸
- 标准路径上下级关系与 N 路径上下级关系必须分开表达，不能共用一个字段
- 团队模式、资源模式属于业务模式，不建议直接写成用户主角色
- 实体门店/自提点优先视为线下站点或合作主体，而不是单纯用户角色
- 邀请码、会员编号、上级 openid 只能作为辅助字段，不能作为核心关系键

### 5.2 下单与支付规则

- 下单主体是用户
- 订单需锁定下单时的价格、等级、优惠、配送方式和履约相关快照
- 支付链路必须独立记录，不能只靠 `orders.paid_at` 判断完整支付事实
- 支付成功后至少要驱动：
  - 支付记录状态更新
  - 订单支付状态更新
  - 库存扣减或库存确认
  - 后续分佣/价差结算触发条件准备
- 支付回调必须幂等，同一回调不能重复改单和重复记账

### 5.3 发货与收货规则

- 快递和自提必须由 `delivery_type` 或等价字段明确区分
- 履约状态必须独立于支付状态
- 自提订单必须能记录站点、核销码、核销时间、核销人
- 代理履约和团队归属必须拆开；实际发货人和买家归属代理是两种不同关系

### 5.4 售后退款规则

- 订单售后退款由用户发起并独立成单
- 退款状态必须独立，至少支持申请、审核、处理中、完成、取消/驳回
- 退款单必须记录退款原因、退款金额、退款数量、凭证图片、审核信息
- 需求文档中的“犹豫期退款”属于资格/合作退款，不应与订单售后退款混为一表

### 5.5 分佣结算规则

- 分佣对象至少包含：标准路径上级、代理履约人、N 路径受益人
- 需求文档中的评级奖/推荐奖如果落地，属于资格类奖励，不应和订单售后退款共域
- 分佣规则不能只支持一种计算方式；必须兼容：
  - 按比例
  - 按固定金额
  - 按等级或规则集差异
  - 按商品覆盖全局规则
- 分佣状态必须独立，至少支持“待生成/冻结/可结算/已结算/已取消”这一类分层
- 分佣触发时点、可结算时点和实际到账时点应分别有字段承接

### 5.6 升级与冻结/犹豫期规则

- 升级申请必须独立记录申请用户、当前等级、目标等级、路径、申请金额、支付方式和审核状态
- 标准路径升级与 N 路径加入/升级可共用申请主表，但应通过 `path_type` 或等价字段区分
- 需求文档中的犹豫期、退款期、评级奖延迟发放等资格规则，若正式落地，需要独立字段承接：
  - `effective_at`
  - `hesitation_start_at`
  - `hesitation_end_at`
  - `qualification_refund_status`
  - `reward_release_status`
- 未确认前，不建议直接写死到 `users.role_level`

### 5.7 审核与后台操作规则

- 平台后台是当前已确定的管理主体
- 审核型业务对象至少包括：
  - 升级申请
  - 退款单
  - 经销商/站点申请
  - 可能的资格奖励发放
- 所有审核类表都应至少包含：
  - `status`
  - `reviewed_admin_id`
  - `reviewed_at`
  - `review_remark`
  - `reject_reason`
- 操作日志应独立保留，不能只靠业务表备注字段承接

## 6. 数据库字段设计约束

### 6.1 主键规范

- 所有核心业务表统一使用 `id` 作为主键
- 业务编号如 `order_no`、`refund_no`、`dealer_no`、`member_no` 仅作为业务编码和查询索引，不作为外键主键

### 6.2 外键命名规范

- 新表和重构字段统一使用显式外键命名：
  - `buyer_user_id`
  - `parent_user_id`
  - `n_leader_user_id`
  - `reviewed_admin_id`
  - `product_id`
  - `sku_id`
  - `order_id`
  - `refund_id`
  - `pickup_station_id`
  - `dealer_id`
- 历史表中已有的 `buyer_id`、`parent_id`、`admin_id`、`claimant_id` 可保留兼容，但不再作为新设计标准

### 6.3 状态字段拆分规范

- 订单相关状态至少拆分为：
  - `order_status`
  - `payment_status`
  - `fulfillment_status`
  - `refund_status`
  - `commission_status`
- 升级/资格相关状态至少拆分为：
  - `application_status`
  - `payment_status`
  - `qualification_status`
  - `reward_release_status`
- 禁止继续使用一个 `status` 字段同时表示“支付成功、已发货、已完成、已分佣”

### 6.4 金额字段规范

- 金额字段统一以 `_amount` 结尾
- 比例字段统一以 `_rate` 结尾
- 当前主线如继续使用“元”作为金额单位，应统一为 `DECIMAL(12,2)` 或更高精度
- 如未来切换为“分”计价，必须整链路同步，不允许同一业务域内混用“元”和“分”
- 百分比在数据库中不存 `"20%"` 这类字符串
- 对于佣金比例，建议统一存 0~1 小数；后台展示层再转换百分数

### 6.5 图片字段规范

- 单主图字段统一用 `image`
- 多图字段统一用 `images`
- 详情图字段统一用 `detail_images`
- 门店/站点图如必须单独命名，可用 `logo_image` 或 `cover_image`
- 不允许在同一业务对象上长期混用 `image_url`、`url`、`image` 作为主图主字段
- 若保留数组式多图，约定 `images[0] = image`

### 6.6 时间字段规范

- 所有表统一保留 `created_at`、`updated_at`
- 业务动作时间按语义拆分：
  - `paid_at`
  - `shipped_at`
  - `verified_at`
  - `completed_at`
  - `reviewed_at`
  - `approved_at`
  - `settled_at`
  - `cancelled_at`
  - `refunded_at`
- 不用模糊字段如 `action_time` 代替多业务时间语义

### 6.7 审计字段规范

- 审核类字段统一使用：
  - `reviewed_admin_id`
  - `reviewed_at`
  - `review_remark`
  - `reject_reason`
- 自动任务生成的业务记录可补充：
  - `generated_by`
  - `source_type`
  - `source_id`

### 6.8 冗余字段兼容策略

- `member_no`、`invite_code` 可保留为展示和绑定辅助字段
- `parent_openid` 仅作为兼容冗余字段，不再作为主关系字段
- `buyer_id` 等历史字段允许短期保留，但新表与重构字段统一采用显式命名
- 图片 URL 旧字段可兼容读取，但新设计不再扩散旧命名

### 6.9 禁止混用的字段设计方式

- 禁止用 `member_no`、`invite_code`、`parent_openid` 作为核心关系外键
- 禁止一个 `status` 字段覆盖支付、履约、退款、分佣等多个生命周期
- 禁止 `agent_id` 同时承担“买家归属代理”和“实际履约代理”两种含义
- 禁止将资格退款和订单售后退款混入同一退款表
- 禁止在分佣规则中只支持比例、不支持固定金额而强行抹平现网逻辑
- 禁止未确认前直接引入 `merchant_id` 作为商品和订单核心外键

## 8. 待确认事项

1. 需求文档中的 `3800 / 3万 / 9万 / 198000` 档位，与当前标准路径 `C1/C2/B1/B2/B3` 的映射关系是什么
2. 需求文档中的“资源模式”是独立用户资格、独立申请类型，还是仅作为奖励规则场景
3. 评级奖 / 推荐奖是否正式进入当前现网数据库设计；若进入，应挂在升级申请域还是独立资格申请域
4. 犹豫期退款是否只针对高档位资格申请，不涉及普通订单；若是，是否需要独立资格退款表
5. 区域经营是仅配置“区域奖励规则”，还是需要独立的区域实体表、区域归属表和区域结算表
6. 实体门店、自提点、服务站点、经销商之间是同一实体的不同类型，还是不同表分别承接
