# 图片素材字段规范

## 当前执行口径

- 历史 COS / 外链图片先保留，不做强制回迁
- 2026-04-08 之后的新后台素材上传，统一走 CloudBase 云存储
- 管理端返回长期正式字段为 `file_id`
- 展示层按需把 `file_id` 转成临时访问 URL
- 默认 `storage-config.mode=managed`，表示新上传不允许回退到本地兜底存储

## 迁移原则

- 老资产兼容读取
- 新资产只进 CloudBase
- 业务表不再把外链 URL 当长期正式字段

## 1. 目标

这份文档定义项目后续图片和素材的长期口径。

核心结论：

**长期正式标识统一为 `file_id`，URL 只作为展示层派生值或迁移期兼容值。**

## 2. 当前存在的问题

当前仓库同时存在这些字段语义：

- `file_id`
- `image_url`
- `url`
- `image`
- `images`
- `avatar_url`

这会导致三个问题：

1. 同一张图在不同模块字段名不同
2. 页面层经常直接读旧 URL 字段
3. 删除、迁移、复用素材时缺少统一主键

## 3. 长期正式字段规范

### 3.1 单图

正式字段：

- `file_id`

兼容字段：

- `url`
- `image_url`

规则：

- `file_id` 是长期主键
- `url` 可以是运行时换出的访问地址
- `image_url` 仅作迁移期兼容，不再作为新标准

### 3.2 多图

正式字段：

- `file_ids`

兼容字段：

- `images`
- `image_urls`

规则：

- 存储层优先保存 `file_ids`
- 页面或接口可派生出 `images` 供旧组件显示

### 3.3 头像

正式字段：

- `avatarFileId`
- `avatarUrl`

兼容字段：

- `avatar_url`

规则：

- 新模型优先 `avatarUrl`
- `avatar_url` 仅保留读取兼容

## 4. 素材库推荐模型

长期建议使用：

- `materials`
- `material_groups`

`materials` 关键字段建议包括：

- `file_id`
- `group_id`
- `name`
- `mime_type`
- `size`
- `width`
- `height`
- `usage_type`
- `created_by`
- `created_at`

`material_groups` 关键字段建议包括：

- `name`
- `code`
- `sort_order`
- `status`

## 5. 业务表如何引用素材

### 5.1 商品

- 主图：`file_ids`
- SKU 图：`file_id`

### 5.2 Banner / 开屏 / 首页内容

- 主图：`file_id`
- 运行时可派生 `url`

### 5.3 用户上传内容

- 评价图：`file_ids`
- 退款图：`file_ids`
- 晒单图：`file_ids`

## 6. 页面展示层规则

页面或接口适配层可以做这些事：

- `file_id` -> 临时访问 URL
- `file_ids` -> `images`
- `avatarFileId` -> `avatarUrl`

但不允许做这些事：

- 把临时 URL 回写成正式主字段
- 继续新增 `image_url` 作为新数据标准

## 7. 图片规格建议

以下规格仍然适用，但它们属于展示建议，不是存储主键规范：

- 首页 Banner：3:2
- 商品主图：1:1
- 缩略图：1:1
- 二维码：1:1

## 8. 当前迁移阶段的兼容策略

根据当前兼容审计，`image_url` 仍在多个区域高频存在：

- 小程序首页、活动页
- 后台内容装修、活动、开屏
- CloudRun 管理服务返回层

当前正确策略：

1. 存储层优先补 `file_id`
2. 接口层同时返回 `file_id + url/image_url`
3. 页面层逐步改为优先读 `file_id` 派生值
4. 迁移完成后再移除旧 URL 字段读取

## 9. 当前必须遵守的规则

1. 新增图片字段时，优先考虑 `file_id` / `file_ids`。
2. 不再新增 `image_url` 作为长期正式字段。
3. 素材删除前必须先确认引用关系。
4. 改商品图、Banner、首页装修、用户上传图时，必须同步检查：
