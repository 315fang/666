# 数据修复脚本索引

## 1. 目的

这个项目已经积累了不少初始化、修复、迁移、审计脚本。

如果没有索引，后续很容易出现两种问题：

- 明明已有脚本，却重复造一个新的
- 不知道脚本影响范围，直接误跑

## 2. 使用原则

1. 先读脚本，再执行，不要靠文件名猜。
2. 涉及数据写入的脚本，先确认运行环境与目标库。
3. 涉及 CloudBase 的脚本，优先使用 `cloud-mp/scripts` 下的标准链路。
4. 修复脚本执行后，要同步更新对应迁移或治理文档。

## 5. 当前推荐使用顺序

### 5.1 做 CloudBase 迁移时

1. `check-cloudbase-foundation.js`
2. `normalize-cloudbase-data.js`
3. `build-cloudbase-import-jsonl.js`
4. `validate-cloudbase-import.js`
5. `audit-legacy-compat.js`

### 5.2 做 MySQL 主系统修复时

1. 先定位是否已有对应 `backend/scripts` 脚本
2. 再确认是否只是临时热修，还是要补长期文档
3. 执行后同步更新：
   - [docs/internals/字段兼容清理清单.md](C:\Users\21963\WeChatProjects\zz\docs\internals\字段兼容清理清单.md)
   - [cloud-mp/docs/CLOUDBASE_MIGRATION_PROGRESS.md](C:\Users\21963\WeChatProjects\zz\cloud-mp\docs\CLOUDBASE_MIGRATION_PROGRESS.md)

3. 不确认环境就直接运行写入脚本。
