# 成长值会员与代理身份融合方案

日期：2026-05-02
状态：文档先行，待按清单实施
适用范围：`cloud-mp` 云函数、小程序、管理后台、会员策略配置

## 1. 结论

当前系统已经不适合继续把“成长值会员”和“代理商体系”做成两套用户心智。

后续应收口为一条主线：

> 用户通过消费、任务、邀请、充值等行为获得成长值和经营指标，成长值与经营指标共同推动 `role_level` 身份成长；`role_level` 决定价格、权益、分销、团队、钱包和升级金猪等能力。

因此：

- 不新增独立的“会员等级”概念。
- 不再让用户同时理解“成长会员等级”和“代理等级”两套等级。
- `role_level` 是用户业务身份的唯一主阶梯。
- `growth_value` 是身份成长进度，不是另一套身份。
- `points` 是积分资产，用于抵扣、任务、运营激励，不参与身份命名。

## 2. 当前依据

### 2.1 文档依据

- `cloud-mp/docs/architecture/current-business-architecture.md` 已说明代理层级用户仍属于 `users` 体系，主字段是 `role_level`。
- `cloud-mp/docs/standards/enterprise-hardening/contracts/user-distribution.md` 已把 `role_level`、`role_name`、`commission_balance`、`goods_fund_balance` 定为用户/分销/钱包正式语义。
- `docs/internals/权限与角色模型总览.md` 已明确“小程序用户角色”和“后台管理员角色”必须分开。
- `docs/internals/分销与佣金逻辑梳理.md` 已说明角色会影响价格、佣金、发货、钱包和升级资格。

### 2.2 代码依据

当前代码已经把成长值纳入代理晋升：

- `cloudfunctions/user/index.js`
  - `deriveEligibleRoleLevel(...)` 会按成长值、销售额、直属成员、充值额计算下一身份。
  - `syncEligibleRoleLevelIfNeeded(...)` 会把可晋升身份同步到 `users.role_level`。
  - `memberTierMeta` 同时返回 `growth_tiers`、`member_levels`、`upgrade_rules` 和 `agent_upgrade_progress`。
- `cloudfunctions/payment/payment-callback.js`
  - 支付成功后发放积分和成长值。
  - 支付后再次计算可晋升身份并更新 `role_level`。
  - 退款链路会扣回成长值和积分。
- `cloudfunctions/distribution/index.js`
  - 分销晋升进度已经读取 `growth_value`。
- `admin-ui/src/views/membership/index.vue`
  - 会员策略页已经同时维护身份层级、成长值门槛、升级规则、积分规则。
- `miniprogram/pages/user/membership-center.js`
  - 会员中心已经同时读取成长档位、身份等级和代理晋升进度。

这说明“成长值会员”和“代理身份”已经事实融合，只是展示、配置和共享代码还没有彻底收口。

## 3. 统一业务模型

### 3.1 唯一身份阶梯

| `role_level` | 统一展示名 | 业务语义 |
| --- | --- | --- |
| `0` | VIP用户 | 注册用户、普通消费用户、未进入代理经营层 |
| `1` | 初级会员 | 已满足基础消费/成长条件，开始拥有基础会员权益 |
| `2` | 高级会员 | 成长值或团队/销售条件达标，拥有更高消费和经营权益 |
| `3` | 推广合伙人 | 进入明确推广经营身份，拥有团队和分销权益 |
| `4` | 运营合伙人 | 更高层级团队经营身份 |
| `5` | 区域合伙人 | 区域级经营身份 |
| `6` | 线下实体门店 | 线下门店身份，历史“店长”应作为别名处理 |

后续所有用户侧展示、后台用户管理、分销中心、会员策略、订单价格、佣金规则，都应优先使用这套阶梯。

### 3.2 三类数值必须分开

| 字段 | 归属 | 能不能命名为等级 |
| --- | --- | --- |
| `role_level` | 身份 | 可以，是唯一身份等级 |
| `growth_value` | 成长进度 | 不可以，只能表示进度和晋升条件 |
| `points` | 积分资产 | 不可以，只能表示积分余额或积分权益 |

### 3.3 资金账户不能融合

以下账户和身份成长有关联，但不能合并：

- `commission_balance`：佣金账户。
- `goods_fund_balance`：货款账户。
- `upgrade_piggy_bank`：升级沉淀金额和解锁金额。

它们可以由身份决定可见性和规则，但不能成为身份字段，也不能相互抵消。

## 4. 需要融合的重复机制

### 4.1 晋升计算重复

当前 `deriveEligibleRoleLevel(...)` 至少存在于：

- `cloudfunctions/user/index.js`
- `cloudfunctions/payment/payment-callback.js`

目标：

- 抽成共享的身份成长引擎。
- 用户查询、会员中心、支付回调、分销进度都调用同一份逻辑。
- 同一用户在“页面看到的可晋升身份”和“支付后真实写入的身份”必须一致。

### 4.2 角色名称重复

当前角色名称散落在云函数、小程序、管理后台多处，并存在 `6 = 店长` 与 `6 = 线下实体门店` 的口径差异。

目标：

- 建立唯一角色字典。
- `线下实体门店` 作为正式展示名。
- `店长` 只作为兼容别名或短文案，不再作为主身份名。

### 4.3 会员策略配置重复

当前后台会员策略同时维护：

- `member_levels`
- `growth_tiers`
- `upgrade_rules`
- `point_levels`
- `point_rules`

目标：

- `member_levels` 表示身份阶梯。
- `growth_tiers` 表示成长值门槛，必须映射到身份阶梯。
- `upgrade_rules` 表示晋升条件。
- `point_rules` 表示积分获取/消耗规则。
- `point_levels` 不再作为主等级展示；如保留，只能作为“积分权益展示档位”。

### 4.4 小程序会员中心心智重复

当前小程序会员中心存在“成长会员”和“代理权益”两张卡。

目标：

- 第一屏只讲“当前身份”和“下一身份进度”。
- 团队、佣金、货款、升级金猪作为当前身份下的权益模块展示。
- 未达到代理经营层时，不把“代理”作为独立心智入口强推给用户。

### 4.5 成长档位展示名重复

`miniprogram/utils/growthTierDisplay.js` 使用“星悦、月华、鎏光、云境、曜世、天冕”等前端覆盖名。

目标：

- 主标题使用统一身份名。
- 装饰名如果保留，只能作为副标题、徽章或视觉主题，不能替代 `role_level` 身份。

### 4.6 积分和成长值历史耦合

当前仍存在若干历史代码把 `points` 与 `growth_value` 同步增减或互相 fallback。

目标：

- 消费抵扣只影响 `points`。
- 身份成长只依据 `growth_value` 和经营指标。
- 退款扣回必须按实际奖励的积分和成长值分别扣回。
- 页面 fallback 可以短期保留，但新代码不得继续把积分当成长值。

## 5. 目标模块边界

### 5.1 身份成长引擎

建议新增共享模块，职责包括：

- 标准化 `role_level`。
- 读取统一角色字典。
- 计算可晋升身份。
- 说明晋升触发类型：成长值、消费、推荐、充值。
- 生成前端可展示的下一身份进度。

禁止职责：

- 不计算佣金。
- 不改钱包余额。
- 不直接处理提现。
- 不做后台管理员权限判断。

### 5.2 会员策略配置

后台继续保留“会员策略”页面，但语义调整为“身份成长策略”。

页面配置分区建议：

1. 身份阶梯：`role_level`、名称、权益摘要。
2. 成长门槛：成长值到身份的映射。
3. 晋升条件：消费、直属成员、充值、成长值。
4. 积分规则：获取、抵扣、任务奖励。
5. 经营权益：团队、佣金、货款、升级金猪开关或摘要。

### 5.3 小程序展示

会员中心建议调整为：

1. 当前身份。
2. 成长值进度。
3. 下一身份条件。
4. 当前权益。
5. 经营权益入口。
6. 积分资产入口。

“代理权益”不再作为和“成长会员”并列的第二套等级，而是当前身份可解锁的经营权益。

### 5.4 管理后台展示

用户管理、财务看板、佣金结算、提现审核等后台页面统一展示：

- 用户身份：`role_name` / `role_level`
- 成长值：`growth_value`
- 积分余额：`points`
- 佣金余额：`commission_balance`
- 货款余额：`goods_fund_balance`

不要再用“会员等级”“代理等级”“积分等级”混合称呼同一个字段。

## 6. 非目标

本次融合不做以下事情：

- 不把小程序用户身份和后台管理员角色合并。
- 不把经销商/分支代理企业资料并入普通会员页。
- 不把佣金账户和货款账户合并。
- 不改变订单支付、退款、提现的财务边界。
- 不新增新的用户等级表。
- 不新增新的“用户需要选择身份”的流程。

- [ ] 后续改动不得新增第二套用户等级心智。

### P2：收口配置和展示

- [ ] 调整后台会员策略页文案为“身份成长策略”。
- [ ] 弱化或隐藏 `point_levels` 的主等级心智。

- [ ] 将“代理权益”改成当前身份下的经营权益模块。
- [ ] 移除或降级成长档位装饰名的主标题地位。

### P3：清历史耦合

- [ ] 审计 `points` 与 `growth_value` 同步增减的位置。
- [ ] 积分抵扣只扣 `points`。
- [ ] 成长值奖励和扣回走成长值专用日志。
- [ ] 清理 `distributor_level`、`agent_level` 等历史字段的写入扩散，只保留兼容读取和必要同步。
- [ ] 更新字段兼容清理文档。

### P4：验证闭环

- [ ] 支付后成长值增加并触发身份晋升。
- [ ] 退款后成长值扣回不破坏已确认的财务流水。
- [ ] 会员中心显示身份与后台用户管理一致。
- [ ] 分销中心晋升进度与会员中心下一身份条件一致。
- [ ] 后台会员策略保存后，小程序和云函数读取同一配置。

## 8. 验收标准

- 当用户打开会员中心时，系统应只展示一条主身份成长路径，而不是并列展示两套等级。
- 当用户支付成功并获得成长值时，系统应通过共享身份成长引擎计算是否晋升。
- 当退款扣回成长值时，系统应按实际奖励记录扣回成长值，不应把积分余额当作成长值。
- 当后台保存会员策略时，系统应把身份阶梯、成长门槛、晋升条件、积分规则分别保存并保持语义清晰。
- 当后台、小程序、分销中心展示用户身份时，系统应使用同一角色字典。
- 当代码需要读取历史 `distributor_level`、`agent_level` 字段时，系统可以兼容读取，但新写入应以 `role_level` 为主。
