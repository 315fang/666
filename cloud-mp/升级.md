# 升级说明

更新日期：2026-04-18

## 1. 本次工作的目标

这轮工作的目标不是新增功能，而是对 `cloud-mp` 的 Markdown 文档做一次集中盘点和收口：

1. 盘点仓库内全部 `.md` 文件
2. 识别哪些是当前项目真相源，哪些是历史快照、阶段报告或参考素材
3. 直接修正文档中已经落后的说明
4. 补一份当前可用的升级记录，说明项目现状和本轮修改范围

## 2. 我对这个项目的当前判断

`cloud-mp` 不是垃圾项目。

更准确地说，它是一个已经做出真实业务面的 CloudBase 电商分销系统：

- 小程序链路完整
- 管理后台链路完整
- 云函数体系完整
- CloudBase 数据基线、导入包和审计脚本体系基本成型

但它也不是“只差润色”的项目。当前核心问题是：

- 文档有明显的阶段漂移
- 一部分旧口径仍在误导协作
- 运行时一致性和工程可信度还有收口空间

## 3. 本次盘点范围

## 5. 本轮文档升级的关键结论

### 结论一：项目当前正式主线是 CloudBase

现在不应该再把这个项目描述成 “Node.js + MySQL 正式运行系统”。

更准确的说法是：

- 正式运行时主线是 CloudBase
- MySQL 是迁移输入和历史资产
- 当前文档如果继续把 MySQL 写成正式主数据源，就是过时文档

### 结论二：文档最大的问题不是“没写”，而是“阶段漂移”

当前仓库的文档问题主要体现在：

- 旧阶段总结还在冒充当前说明
- 一些导航页指向旧路径和旧入口
- 一些审计报告还是旧脚本生成的结果
- 运行说明和当前代码边界没有完全同步

### 结论三：这个项目值得继续修，不值得重写

文档收口之后，这个项目会更接近一个“工程面偏脆但业务价值明确”的项目，而不是一个“看起来什么都写了其实没人敢碰”的项目。

## 6. 本轮没有大动的部分

这轮没有逐个重写所有专项审计和历史快照，原因是：

- 很多文件属于时间点快照，本来就不适合改写成长期真相源
- 一些文件属于运行证据或阶段报告，保留其历史属性比硬改更合理

后续如果要继续清理，建议优先处理：

- 仍带旧阶段完成度口径的根目录说明
- 重复导航页
- 一次性交付总结
- 容易误导协作的旧审计结论

## 7. 下一步建议

1. 继续把文档主入口收口到少数几份核心文件
2. 把历史阶段总结和当前说明继续分层
3. 结合代码修复以下高风险项：
   - `collectionPrefix` 与精确写入路径不一致
   - `admin-api` 冷启动 fail-open
   - `saveCollection()` 整集合写回风险
   - MySQL 兼容路径继续留在主分支中的误导性
