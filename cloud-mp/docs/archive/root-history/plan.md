## User Requirements

- 解释商品详情页里黑色 `ok` 提示的含义，并将这类无意义反馈替换为用户可理解的状态提示。
- 当用户已参与砍价或拼团时，页面必须明确告诉用户当前状态，并提供继续查看入口，不能只提示“已参与”却不给去向。
- 商品详情页需要让用户清楚区分“立即发起”和“继续查看”两类动作，避免用户误判当前所处阶段。
- 需要补强用户回找自己活动的路径，让用户能从常用页面快速进入自己的砍价或拼团记录。

## Product Overview

当前商品详情页已支持普通购买、拼团购买、砍价购买，但活动参与后的承接不清晰。方案应把商品详情页、活动专区、我的页面、订单页面串成一条清晰路径，让用户能看懂自己是否已参与、活动是否还在进行、下一步应该去哪里。

## Core Features

- 商品详情页的活动状态提示与主按钮文案优化
- 已参与砍价时自动恢复到砍价详情或明确跳转到“我的砍价”
- 为拼团与砍价补充稳定的“我的活动”入口
- 在订单或活动页补充“查看进度/查看详情”承接按钮
- 活动专区对“我的拼团/我的砍价”状态展示加强，减少用户寻找成本

## Tech Stack Selection

- 小程序前端：原生微信小程序页面结构，基于 `miniprogram/pages/**` 的 JS、WXML、WXSS 组织
- 请求层：复用 `miniprogram/utils/request.js` 与 `miniprogram/utils/cloud.js`
- 后端：复用 CloudBase 云函数 `cloudfunctions/order/**`、`cloudfunctions/payment/**`
- 数据承接：复用现有 `group_orders`、`slash_records`、`orders` 数据模型与字段风格

## Implementation Approach

### High-level Strategy

本次改造以“复用现有页面、补全续接链路”为主，不新建复杂活动系统。核心做法是：前端统一把“发起失败/已参与/继续查看”分开处理，后端为重复参与场景返回可恢复详情标识，商品详情页、我的页、专区页、订单页共同提供活动回找入口。

### Key Technical Decisions

- **砍价优先做自动恢复**：现有 `slash_records` 在发起时就已生成，最适合在“已发起过砍价”时直接返回 `slash_no` 并跳转详情。
- **拼团优先对齐现有支付链路**：当前 `group_no` 在支付回调后才真正稳定建立，商品详情页不应继续依赖“发起即返回 group_no”的错误假设，应复用现有下单确认与支付后查看进度链路。
- **接入现有 `activityResume.js`**：避免新增第二套恢复判断逻辑，统一解析“已有活动/继续查看”返回结构，减少分散分支。
- **入口补强而非全量重构**：优先在已有页面增加显式入口与状态按钮，避免引入新的营销中心模式。

### Performance & Reliability

- 商品详情页不建议在首屏额外拉取多次“我的活动状态”查询，优先在用户点击活动主按钮时处理恢复，减少无必要请求。
- 砍价恢复处理为单次点击单次请求，复杂度为 O(1) 前端分支加一次已有后端查询。
- 拼团承接尽量复用已有订单与支付后回调，不新增预创建团的复杂事务，降低状态错乱风险。
- 后端返回结构需保持向后兼容：旧前端仍可读 `message`，新前端额外消费 `existing`、`slash_no`、`group_no` 等字段。

## Implementation Notes

- 继续复用 `miniprogram/utils/activityResume.js`，不要在页面内硬编码“已参与”关键词判断。
- `ok` 不应再作为任何用户可见提示；前端必须优先判断关键返回字段，再决定文案或跳转。
- 拼团、砍价的入口改造应限制在活动相关页面，避免波及普通购买与购物袋逻辑。
- 若补齐砍价订单关联，支付成功后的状态回写应放在已有支付后处理链路中，避免在前端猜测购买完成状态。
- 订单页新增活动入口时要有安全兜底：无 `group_no` 时只展示说明，不触发错误跳转。

## Architecture Design

### Current-aligned Structure

- **商品详情层**：选择购买方式、发起活动、处理恢复跳转
- **活动详情层**：展示拼团/砍价进度与分享状态
- **我的活动层**：聚合用户自己的拼团与砍价记录
- **订单辅助层**：承接支付后回看活动详情
- **后端互动层**：为前端返回可恢复活动标识与活动关联字段

### Data/Flow Adjustment

- 砍价：商品详情页发起 -> 后端若新建成功返回 `slash_no` -> 进入砍价详情；若已存在进行中记录，返回可恢复信息 -> 直接进入砍价详情
- 拼团：商品详情页发起 -> 进入订单确认与支付 -> 支付回调建立或加入 `group_orders` -> 订单页进入拼团详情
- 我的页/专区页：提供稳定入口 -> 进入 `我的拼团` 或 `我的砍价` -> 再进详情页

## Design Direction

在保留现有品牌气质的前提下，采用“轻奢感、状态优先、路径直达”的移动端改造思路。重点不是做花哨视觉，而是让用户一眼看懂自己是否已参与、活动进行到哪一步、下一步去哪里。

## Screen Planning

### 1. 商品详情页

- 顶部购买方式区保留“普通购买 / 拼团购买 / 砍价购买”，但活动模式下增加状态提示条。

- 在“我的功能”区域增加“我的拼团”“我的砍价”两个直达项。

- 默认突出“我的记录”入口与当前状态标签。
