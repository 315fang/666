# 企业级规范化收口需求

日期：2026-04-14

## 1. 背景

当前项目已经具备业务能力，但工程契约、字段语义、文档入口、验证闭环尚未达到企业级交付标准。  
最核心的问题不是功能缺失，而是：

- 同一业务对象没有完全统一的正式字段
- 小程序、后台、云函数之间存在隐形兼容层
- 代码修复缺少稳定的修复文档和验收口径
- 发布前缺少清晰的本地与预发双门槛

## 2. 业务目标

1. 建立单一、权威、可维护的规范化入口。
2. 固定三条主链和请求层的正式契约。
3. 固定每一轮代码修复的文档模板和验证流程。
4. 建立本地阻断门槛、预发阻断门槛、发布前检查表。
5. 在不破坏当前外部接口的前提下，逐步移除深层兼容债。

## 3. 非目标

以下内容不属于本轮交付目标：

- 不新增业务功能
- 不变更现有外部 URL、云函数 `action`、后台 `/admin/api/*` 路径
- 不执行线上部署
- 不把全仓一次性重写
- 不在没有证据的情况下删除所有历史兼容字段

## 4. 主要使用者

- 项目负责人：需要知道项目何时能审查、何时可发布
- 开发者：需要知道正式字段、修复顺序、禁止事项
- QA / 审核者：需要知道如何验证是否通过
- 运维 / 发布执行者：需要知道发布前检查项和回滚路径

## 5. 用户故事

1. 作为项目负责人，我需要一套总规范入口，以便判断当前项目离企业级交付还差什么。
2. 作为后端或云函数开发者，我需要明确的正式字段和兼容策略，以便不再新增别名和隐形逻辑。
3. 作为前端或小程序开发者，我需要知道页面应该消费哪些正式字段，以便不再本地猜状态、金额和支付方式。
4. 作为 QA，我需要一套固定的本地和预发门槛，以便判断这轮修复是否真的完成。
5. 作为发布执行者，我需要一套发布与回滚文档，以便在上线前后按统一流程操作。

## 6. 约束

- 正式字段必须成为唯一真相源
- 旧字段只能由正式字段推导
- 兼容逻辑只能存在于 contract / normalizer / adapter / assembler
- 任何数据修复逻辑都必须配套 repair 脚本或 repair 文档
- 大入口文件不得继续吸收新职责
- 文档必须引用现有审计结果，而不是重复制造第二份真相

## 7. EARS 验收条件

### R1 文档入口

当工程师开始规范化收口工作时，系统应提供单一入口文档，明确范围、阶段、真相源和使用顺序。

### R2 正式契约

当工程师修改订单、用户分销、配置内容或请求传输链路时，系统应提供对应的正式契约文档，明确正式字段、兼容字段和禁止新增项。

### R3 修复文档

当一轮代码修复完成时，系统应产出一份统一格式的修复文档，说明问题、根因、修复方案、兼容策略和验证结果。

### R4 本地验收

当开发者声称某轮收口“已完成”时，系统应要求其通过所有本地阻断门槛，否则不得视为完成。

### R5 预发验收

当项目进入上线准备阶段时，系统应要求其通过预发和真机阻断门槛，否则不得进入发布执行。

### R6 最终交付

当项目进入审查或交付阶段时，系统应提供最终交付文档，明确已完成范围、剩余技术债、上线阻断项和发布前依赖。

### R7 接口兼容

当系统执行规范化收口时，系统应默认保持现有外部接口兼容，除非有单独批准的迁移方案。

### R8 真相一致

当同一条业务链被小程序、后台、云函数和数据层共同消费时，系统应保证其字段语义一致，并有自动化证据支撑。
